# CHERENKOV QA — System Design

**Authority:** [docs/STATUS.md](STATUS.md) · [docs/PHASE_PLAN.md](PHASE_PLAN.md)

This document serves as the high-level system design and architectural mapping for the Cherenkov QA engine. It aligns with the "Clean Architecture" (Ports & Adapters) principles detailed in `docs/adr/ADR-004-clean-architecture.md`.

## 1. High-Level Architecture (The Reality Engine)
Cherenkov operates as an autonomous, localized, high-fidelity API conformance generator. The platform is designed strictly to avoid lock-in (D7 Invariant: Suggest-Only, ejective tests). 

The flow spans across multiple isolated layers:
```
OpenAPI Spec → Ingestion & Parsing (Depth-1) → Scenario Planning (Deterministic) → Generation (VLM/LocalAI/Ollama) → 6-Gate Review (AST, tsc, assertions) → Persistent Tests
```

## 2. Ports and Adapters (Clean Architecture)
To ensure isolation, testability, and portability, the entire `cherenkov/` codebase is modeled using Ports and Adapters.

### Domain Layer (`cherenkov/domain/`)
- Contains pure business logic.
- **Rules:** No file I/O, no network calls, no infrastructure dependencies. Pydantic models govern strict types (e.g., `DivergenceReport`, `TestScenario`).

### Ports (`cherenkov/ports/`)
- Abstract base classes defining system interfaces ("The What").
- Examples: `DatabasePort`, `LLMPort`, `NotificationPort`.

### Adapters (`cherenkov/adapters/`)
- Concrete implementations of Ports ("The How").
- Examples: `SQLiteRepository`, `OllamaClient`, `SlackNotifier`, `JiraExporter`.
- **Rules:** If a dependency changes (e.g., swapping Ollama for LocalAI), only the adapter changes.

### Use Cases (`cherenkov/use_cases/`)
- The orchestration layer that glues Domains to Adapters.
- Examples: `GenerateTestsUseCase`, `ValidateDriftUseCase`.

### Delivery / API (`cherenkov/web/`, `cherenkov/cli/`)
- FastAPI endpoints, SSE streaming servers, and Typer CLI commands.
- These modules map HTTP/CLI requests directly to the Use Cases.

## 3. Modularity & The 16 Horizons
The system scales horizontally by bolting on isolated adapters, preserving the core invariant:
- **Phase 1-8 (Core):** CLI runner, Playwright generator, Prism verification, K8s gating, Desktop Tauri 2 host.
- **Phase 9-16 (Extended):** CI/CD hooks (GitHub Actions), Tiered Notifications (Slack/Teams/Jira/PagerDuty), Protocol support (GraphQL/gRPC/AsyncAPI), and VS Code IDE extensions.

## 4. Design Invariants
1. **D7 Suggest-Only:** The engine can detect drift and *suggest* healing assertions, but it will **never** automatically rewrite a user's test suite or break CI/CD blindly.
2. **Ejectability:** Tests generated by Cherenkov are pure `@playwright/test`. They can be run independently of Cherenkov.
3. **Spec as SSOT:** Expected HTTP statuses are derived deterministically from the OpenAPI spec, not from LLM hallucinations.
